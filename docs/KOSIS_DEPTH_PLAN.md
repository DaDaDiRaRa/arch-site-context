# KOSIS 깊이 격차 독자 재구축 계획

> 작성 2026-06-29. 경쟁 외부 프로젝트(arch-analysis-mcp, v3.0)의 강점인 "KOSIS 196항목 × 20+ 용도"를
> **카피 없이 독자 재구축**해 터읽기의 최대 약점(KOSIS ~7지표·용도 3개, §8.6·DEFERRED D4)을 메운다.

---

## 0. 전제 (법적·전략적)

- **경쟁사 matrix/코드는 가져오지 않는다** — 접근 불가 + 경쟁사 IP. 참고하는 것은 *개념*(어떤 용도에
  어떤 인구·경제 지표가 의미 있는가)뿐이며, 그 설계 판단은 우리가 독립적으로 내린다.
- **KOSIS 통계표 자체는 공공 오픈데이터** → 카탈로그 API로 우리가 직접 발굴. 결과적으로 수작업보다
  체계적이고 재현 가능(절대 원칙 1 "추출·해석하지 않는다", 5 "출처·기준 명시").
- **목표는 숫자(196) 맞추기가 아니다.** *시군구 해석 가능 + 출처·기준연도 명시 + 용도 매핑된* 지표를
  넓히는 것. 허수 카운트(시도전용·미확정)는 거부한다(§8.6 정직성).

## 1. 왜 가능한가 (증명 완료 2026-06-29)

KOSIS 카탈로그 API가 작동함을 실호출로 확인:

```
GET https://kosis.kr/openapi/statisticsList.do
    ?method=getList&apiKey=...&format=json&jsonVD=Y&vwCd=MT_ZTITLE&parentListId=A_7
→ HTTP 200, "인구" 주제 하위 시군구 통계표 9개 (TBL_ID/ORG_ID/TBL_NM 포함)
  예: DT_1B040A3 행정구역(시군구)별·성별 인구수 / DT_1B04006 행정구역(시군구)별·1세별 주민등록인구
```

즉 경쟁사가 수작업으로 찾은 통계표를 우리는 **프로그램으로 전수 크롤**할 수 있다.

**Phase 1 시범 실행 결과 (2026-06-29, `scripts/mine_kosis_catalog.py --topics A,I1`):**
- 단 2개 주제(인구 A + 주거 I1) → 폴더 325개 크롤, 통계표 2,887개, **시군구 후보 728개**.
- 14개 주제 전체면 수천 개 예상 → **경쟁사 "196"은 매칭이 아니라 압도 가능 수준**.
- 결론: 병목은 *발굴*이 아니라 Phase 2(어떤 표가 시군구 값으로 깨끗이 해석되는가)·Phase 3(용도별 큐레이션).
- 주의: "행정구역" 필터가 읍면동 표도 포함 → Phase 2 검증에서 시군구 해석 가능분만 정밀 선별.

## 2. Phase 구성

### Phase 1 — KOSIS 카탈로그 마이너 🎯 (키스톤, 증명 완료)

`scripts/mine_kosis_catalog.py`:
- `statisticsList.do(method=getList)`로 주제 트리(`vwCd=MT_ZTITLE`)를 재귀 크롤 →
  인구·가구·주택·사업체·복지·문화·교육·노동·소득 등 주제별 통계표 전수 수집.
- `TBL_NM`에 "시군구"/"행정구역" 포함 표만 1차 필터 → `out/kosis_catalog.json` 적재.
- 산출물 = **시군구 해석 후보 통계표 마스터 목록** (경쟁사 196의 원천 풀).
- 위험 0: 읽기 전용 스크립트, 기존 기능 무영향. KOSIS 분당 호출 제한 → 캐시·슬립.

### Phase 2 — 항목코드 자동 해석 (§8.6의 진짜 난관)

각 후보 표를 `itmId=ALL, objL=ALL`로 1회 질의 → 응답에서 자동 역추출:
- **itmId**(지표 항목)·**분류코드 objL**(연령밴드·가구원수 등)·**단위**·**기준연도**.
- **지역코드 체계**(reg=행안부 / census=인구총조사 역추출, §8.6) 자동 판별.
- **셀상한(err31)·시도전용 표** 자동 분류·제외.
- 터읽기가 §8.6에서 손으로 한 "응답 역추출"을 자동화·일반화.
- 산출물 = 표마다 "이 파라미터로 호출하면 시군구 값이 나온다"는 검증된 호출 스펙.

### Phase 3 — 용도→지표 매핑 (우리의 IP·진짜 차별점)

20+ 용도(주거·상업·의료 + 숙박·공공·산업·복합·교육·문화·근린생활 …) 각각에 건축가 관점에서
의미 있는 지표를 선정 → Phase 2의 검증 표와 연결 → `matrix.json` 생성/확장.
- 이건 코드가 아니라 **설계 판단** → 경쟁사와 독립, 터읽기 §2 "재사용 가능한 팀 자산".
- 우선순위(1필수/2권장/3선택)·method(direct/age_share/ratio/…)·region_scheme 부여.

### Phase 4 — 검증 하니스

`verify_apis.py` 패턴으로 생성 matrix 전 항목을 영등포·강남·부산 등 다지역 실호출 →
시군구 값이 실제로 나오는지 게이트. **통과분만 `method≠unconfirmed`**. CI/주기 점검 자산화.

### Phase 5 — 통합·노출

- 밀도/사업체수 등은 §8.6 메모대로 **수급진단에 연결**(면적·인구 결합) — 모드 A raw 전국비교는 무의미.
- 프론트 모드 A 용도 드롭다운 3 → 20+ 확장.
- 막혔던 **DEFERRED D4(미확정 지표)·로드맵 P16(용도 확장) 동시 해결**.

## 3. 기대 효과

| | 현재 | 계획 후 |
|---|---|---|
| KOSIS 확정 지표 | ~7 | 카탈로그 발굴분 (Phase 1 실측으로 확정) |
| 건물 용도 | 3 (주거·상업·의료) | 20+ |
| 차별점 | 수급진단(A×B) | 수급진단 **+ KOSIS 깊이** (경쟁사 강점 흡수) |

경쟁사는 KOSIS 깊이 1개 축, 터읽기는 그 깊이 **＋ 수급진단·멀티모달·지도·부동산** →
깊이를 흡수하면 경쟁사 우위 항목이 사라지고 터읽기 고유 기능만 남는다.

## 4. 진행 원칙

- 한 번에 한 Phase. Phase 1 실측 결과로 "196이 현실적 목표인가"를 데이터로 먼저 판정.
- 추정으로 채우지 않는다 — 검증 실패 표는 `unconfirmed`로 정직 표기(절대 원칙 3).
- matrix는 JSON만 수정(절대 원칙 7) — 코드 변경 최소.

### Phase 2 발견 (2026-06-29) — "완전 자동화"의 벽

- **메타 API가 키** — `statisticsData.do?method=getMeta&type=ITM|PRD`:
  - `type=ITM` → 항목코드·이름·단위 (표당 수백 개). `type=PRD` → 주기(년/5년/월)·최신연도.
  - **이건 완전 자동화됨** → 항목·단위·최신성은 표 전체에서 기계적으로 추출 가능.
- **시군구 해석 검증은 자동화 불가(범용)** — 표마다 분류차원(objL2·objL3 …)이 달라
  범용 질의(`objL1=영등포, itmId=ALL`)는 err20/21(필수 분류 누락)이 난다. **우리가 프로덕션에서
  쓰는 DT_1B04005N 조차** 이 범용질의로는 ✗(objL2=연령밴드 필수). 즉 region 해석은 **표별 수동 확정** 필요.
  - 이게 §8.6의 본질이자 경쟁사 "196"의 실제 노동 — 표 196개를 손으로 튜닝했다는 뜻.
- **재설계된 효율 경로**:
  1. Phase 2(자동): getMeta 로 후보 표의 항목·최신연도 일괄 추출 → **최신(2015+)·건축관련 표 리치 숏리스트**.
  2. Phase 3(수동, 유한): 숏리스트에서 고가치 표 ~30-50개만 골라 objL 파라미터를 손으로 확정
     (=경쟁사가 한 일을, 우리는 카탈로그+메타로 *대상을 좁혀* 빠르게). 검증된 것만 matrix 편입.
- **즉시 쓸 수 있는 단순표(1차원 시군구)**: DT_1B040A3(시군구 성별 인구)·DT_1B040B3(시군구 세대수)·
  DT_1B26001_A01(시군구 인구이동) 등은 범용질의로 바로 해석됨 → 우선 편입 후보.
