SPEC — 터읽기 (arch-site-context)

역추론 명세 v1.0 2026-06-29 FastAPI + React Python 3.11 GCP Cloud Run
이 문서는 코드베이스를 읽고 목적·로직·구조·한계를 역추론한 명세다. 불확실한 부분은 추정 으로 표시한다. 최종 판단 권위는 CLAUDE.md 및 실제 코드다.

1 목적

주소 한 줄 → 건축 대지 주변 맥락 보고서 자동 생성.

건축가가 설계 착수 전 반나절 이상 걸리던 '대지 주변 인문·생활 조사'를, 실제 공공 API에서 끌어온 수치로 자동화한다. 최종 설계 판단은 사람이 한다.

탭 A
지역 통계
주소 + 용도 → 인구·가구 통계 + 시사점 + Claude 문단
탭 B
주변 시설
반경별 시설 목록·개수 + 위성 PNG (TMAP 등시선)
탭 C ★
수급 진단
인구 수요 × 시설 공급 교차 → 부족/과잉 신호 (참고)
탭 D
후보지 비교
2~5개 주소 → A·B·C 나란히 비교 (종합점수 없음)
탭 E
물어보기
데이터 위에서만 답변. 데이터 밖이면 "확인 불가"로 멈춤

사용자 페르소나 추정

  • 1차 사용자: 공모(설계경기)·실무 설계를 준비하는 건축가
  • 목적 맥락: 대지분석 보드의 '인문·생활' 파트를 채우는 용도
  • 보완 도구: 일조·바람·사업수지·규제 해석 도구와 병행 사용 (이 앱은 그쪽을 만들지 않음)

2 핵심 설계 원칙 7가지

단순 가이드라인이 아니라 코드 전체에 구조적으로 박혀 있다.

1
실제 API에서만 — AI 기억·추정 금지
KOSIS·카카오·에어코리아 등 직접 호출. 값 없으면 None, 추정 없음
2
수치는 코드, 표현만 AI
facts[]·implications[]은 규칙·KOSIS가 만든다. Claude는 draft_paragraph 서술만
3
확인 불가 하드블록
데이터 없으면 ErrorBlock 반환 or notes 기록, 빈 추정값 생성 금지
4
출처·기준 명시
모든 Fact에 source_tbl, year 필드. "○○구 기준" 문단 필수
5
판단은 사람
모든 Implication·Diagnosis가 tag:"참고". 종합점수 없음
6
모델은 Claude 하나
narrative.py, ask.py 전부 claude-opus-4-8. 교차검증 없음
7
설정은 JSON
matrix.json, implications.json, supply_demand.json — 코드 수정 없이 건축가가 편집

4 엔드포인트 명세

POST /analyze 지역 통계 (모드 A)

입력
{ "address": "서울특별시 영등포구 여의대로 24", "use_type": "주거", "year": null }
처리 흐름
resolve_address(address) → sgg_code 없으면 ErrorBlock(NO_REGION_CODE) → collect_facts(sgg_code, use_type) // KOSIS + 에어코리아 → collect_common_facts(sido, sigungu) // _common 항목 → derive_implications(facts, use_type) // implications.json 룩업 → compose_narrative(facts, implications) // Claude 1회 or 규칙 폴백
출력 — RegionStat
{
  "region": { "name": "영등포구", "code": "11560", "resolution": "시군구" },
  "year": 2024, "use_type": "주거",
  "facts": [
    { "item": "1인가구비율", "value": 38.2, "national_avg": 33.4,
      "unit": "%", "source_tbl": "DT_1JC1511", "year": 2024 }
  ],
  "implications": [
    { "text": "소형 평형·공유공간 검토", "basis": "1인가구비율", "tag": "참고" }
  ],
  "draft_paragraph": "영등포구는 1인가구 비율이 전국 평균보다 높아...",
  "source": "ai",   // "ai" | "rule_based_fallback"
  "notes": []
}

ErrorBlock 조건: ADDR_UNRESOLVED · NO_REGION_CODE · NO_DATA

POST /facilities 주변 시설 (모드 B)

처리 흐름
resolve_address(address) → 카카오 키워드 검색 (적응 분할, 45건 상한 회피) → OSM Overpass 보완 (공공시설) → VWorld 검색 보완 (경로당·노인복지관 등 비상업 시설) → 중복 제거 (이름+좌표 6자리) → 하버사인 거리 → 반경 밴드 분류 → 거리순 정렬

counts 누적 방식: 500m 시설은 500/1000/2000 밴드 전부 집계 (누적 포함).

POST /facilities/map 위성 PNG 합성

추가 필드: "basemap": "vworld", "isochrone": true

등시선 생성 알고리즘
16방향 × 3시간(5·10·15분) = 48회 TMAP 병렬 호출 (ThreadPoolExecutor) → 각 경로에서 목표 시간(초) 도달 지점 보간 → 꼭짓점 → 꼭짓점 연결 → 폴리곤 → PIL 채움 (초록·노랑·주황, 외→내) → 실패 방향 제외 (graceful) → 전부 실패 시 반경원 폴백

POST /diagnose 수급진단 ★간판

판정 로직
수요 레벨:
  value > national_avg + margin  →  "높음"
  value < national_avg - margin  →  "낮음"
  그 사이                         →  "평이"
  national_avg 없음              →  "불명"

공급 레벨 (반경 비례, primary):
  scale  = (radius / 1000)²
  count ≤ supply_low_max × scale   →  "적음"
  count ≥ supply_high_min × scale  →  "많음"
  그 사이                           →  "보통"

공급 밀도 (보조 참고, 판정에 미사용):
  density_per_10k  = count / (시군구총인구 / 10_000)
  vs_national_pct  = density_per_10k / national_density_per_10k × 100
출력 — DiagnoseResult (진단 1개 예시)
{
  "name": "보육시설 수급",
  "demand": { "item": "유소년인구비율", "value": 8.3, "national_avg": 10.3, "level": "낮음" },
  "supply": {
    "kinds": ["어린이집", "유치원"], "count": 23, "level": "많음",
    "density_per_10k": 0.62, "national_density_per_10k": 7.7, "vs_national_pct": 8,
    "capacity": 2785, "capacity_scope": "영등포구"
  },
  "signal": "수요 낮음·공급 많음",
  "note": "유소년인구비율 8.3%(전국 10.3%) · 반경 1000m 23개 → 과잉 여부 점검",
  "tag": "참고"   // 항상 "참고" — 판단은 사람 (원칙 5)
}

POST /compare 후보지 비교 (P9)

  • 2~5개 주소 제한
  • 후보지별 격리 실패: 한 주소 오류 → CompareSite.error 기록, 나머지 계속
  • 종합점수 없음 — 컬럼 정렬만 (판단은 사람, 원칙 5)
  • 같은 시군구 코드이면 KOSIS 캐시 재사용 → 추가 API 호출 없음

POST /ask 물어보기 (P10)

처리 흐름
gather_bundle(address) → facts + counts + diagnoses // A·B·P11 한 번에 → answer_grounded(bundle, question) // Claude: 데이터 위에서만 답 answerable=false 시: "확인 불가:" 로 시작 → (web=true) answer_web(question) // web_search 서버툴 폴백, 최대 4회 루프

source 값: "ai" · "ai_web" · "no_data" · "ai_unavailable"

5 데이터 계약 요약

공통 패턴

  • Fact: 항상 { item, value, national_avg, unit, source_tbl, year } — 수치와 출처 분리 불가
  • Implication: 항상 { text, basis, tag:"참고" } — '좋다/나쁘다' 단정 없음
  • ErrorBlock: { code, message } + HTTP 422 — 추정·빈 값 반환 없음
  • notes[]: 모든 응답에 포함 — 실패·우회·주의 사항 정직 기록

SupplySignal 필드 (P11 확장 후)

Python
class SupplySignal:
    kinds:   List[str]
    count:   int          # 반경 내 실제 개수 (primary 판정)
    radius:  int
    level:   str          # "적음" | "보통" | "많음" (count 기반)

    density_per_10k:          Optional[float]  # count / (시군구총인구 / 10000) (참고)
    national_density_per_10k: Optional[float]  # 전국 만명당 기준값 (참고)
    vs_national_pct:          Optional[int]    # 전국 대비 % (참고)

    capacity:       Optional[int]   # 시군구 어린이집 정원 (반경 개수와 단위 다름)
    capacity_scope: Optional[str]

11 한계 (실측·추론)

데이터 공간 해상도

  • KOSIS 통계는 시군구 평균 — 대지 고유값 아님
  • 읍면동 단위 통계는 대부분 KOSIS에 없거나 별도 코드 체계
  • 반경 내 인구 계산 불가 → 공급 판정 분모 = 시군구 전체 인구 (공간 불일치)

수급진단 임계값

  • supply_low_max·supply_high_min은 휴리스틱 (JSON에 근거 출처 없음)
  • 전국 밀도의 분모·분자가 다른 공간 단위 → 절대 비교는 의미 제한
  • 상업·문화 시설(수요 proxy 없음) 수급진단 미편입 추정

시설 검색

  • 카카오 45건 상한 → 매우 밀집 지역 누락 가능
  • 카카오 키워드 오매칭·폐업 정보 지연
  • VWorld 검색: 경로당·노인복지관·마을회관에만 국한
  • OSM 지역별 갱신 주기 불균일 추정

등시선

  • 16방향 고정 → 실제 보행 경계와 오차
  • 경로 없는 방향 꼭짓점 제외 → 폴리곤 들쭉날쭉
  • TMAP 호출 제한 도달 시 부분 실패 추정

지역 제한

  • 서울 생활인구: 서울시 전용 — 타 시도는 null
  • KOPIS 이름 필터 → 동명 시군구 오매칭 추정
  • 실거래 없는 기간·지역에서 0건 반환

KOSIS 미확정 지표

  • 인구밀도 — 면적 테이블 결합 필요
  • 사업체수·종사자수 — 40,000셀 초과, 코드 체계 별도
  • 주택보급률·주간인구지수 — tblId 미확정

앱이 안 만드는 것

  • 물리 시뮬레이션 (일조·바람·그림자)
  • 재무 분석 (사업수지·NPV·분양가)
  • 규제 해석 ("뭘 지을 수 있는가")
  • 설계 안 생성

기술 부채

  • httpx + starlette.testclient deprecation (httpx2 필요)
  • GCS 캐시 + TTL 미구현 (멀티 인스턴스)
  • PNG 저장 → /tmp 의존 (재시작 시 소실) 추정
  • JUSO 운영키 미교체 (현재 dev키)

14 설계 결정 이유 추정

supply_low_max·supply_high_min을 JSON에 둔 것
건축가가 현장 피드백으로 임계값을 조정할 수 있게 — 코드 배포 없이 수정 가능 (원칙 7)
공급 판정을 밀도가 아닌 개수 기반으로 한 것
분모(시군구 전체인구) ≠ 분자(반경 내 시설)의 공간 불일치. 절대 오류보다 상대 오류가 작은 개수 기반이 더 안전
Claude 모델을 하나만 쓴 것
정답이 정해진 데이터에서 다중 모델 교차는 잡음만 늘림 (원칙 6)
함의(implications)를 LLM이 아닌 규칙으로 만든 것
LLM이 새 수치를 생성하지 않도록 — 규칙은 재현 가능하고 감사(audit) 가능 (원칙 2)
수급진단에 종합점수를 두지 않은 것
"좋다/나쁘다" 단정은 설계 판단 — 그 판단은 건축가가 해야 함 (원칙 5)
OSM·VWorld를 보충 소스로 쓴 것
카카오 45건 상한 + 비상업 시설(경로당 등) 누락 보완. 각 독립 실패 격리로 안전
서울 생활인구만 서울 전용인 것
SEOUL_API 자체가 서울시 데이터만 제공 — 다른 시도엔 해당 API 없음
TMAP 등시선에서 반경 2.5배 오프셋을 쓴 것
직선 거리보다 실제 도로 우회 경로가 길기 때문. 2.5는 도시 도로 굴곡율 경험치 추정
